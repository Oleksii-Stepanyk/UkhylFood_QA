# Стратегія модульного тестування (Unit Testing Strategy)

Цей документ визначає суворі правила та стандарти для написання модульних тестів у проекті `KSE.DistributedSystems.CourierService` з використанням **xUnit** та **Moq**. Мета стратегії — забезпечити стабільність, зрозумілість, легкість підтримки та високу якість тестування, уникаючи поширених антипатернів, таких як надмірне використання моків (over-mocking) та "магічні значення" (magic values).

## 1. Архітектура та структура тестів
- **Розташування:** Тести повинні знаходитися в проекті `KSE.DistributedSystems.CourierService.Tests`.
- **Файлова структура:** Тестові класи повинні бути логічно розділені за функціоналом (наприклад, `CourierRegistrationTests.cs`, `CourierLocationTests.cs`, `CourierAvailabilityTests.cs`). Один великий файл на кшталт `CourierServiceUnitTests.cs` заборонений.
- **Найменування класів та методів:** 
  - Класи: `[Назва компонента]Tests`
  - Методи: `[НазваМетоду]_[Сценарій]_[ОчікуванийРезультат]` (наприклад, `RegisterCourier_WithValidData_ReturnsCorrectReadDto`).
- **Патерн AAA:** Кожен тест повинен суворо дотримуватися розділення на блоки `Arrange`, `Act`, `Assert`, відокремлені коментарями або порожніми рядками.

## 2. Правила використання Moq (Mocking Rules)
- **Коли використовувати:** Тільки для ізоляції зовнішніх залежностей (база даних / репозиторії, зовнішні API, шини повідомлень `MassTransit`, логери).
- **Уникнення Over-mocking:** 
  - Не мокайте внутрішні деталі реалізації, такі як точні параметри специфічних команд Redis (`CommandFlags`), якщо це жорстко прив'язує тест до конкретної реалізації кешу.
  - Використовуйте `It.IsAny<T>()`, коли точне значення параметра не впливає на бізнес-логіку поточного сценарію.
  - Використовуйте `Times.Once()`, `Times.Never()` лише для перевірки критичних сайд-ефектів (наприклад, виклики бази даних або публікація подій).
- **Метрики та OpenTelemetry:** Метрики слід ігнорувати за допомогою фіктивних трейсерів (`TracerProviderBuilder().Build().GetTracer("Dummy")`), оскільки вони не є частиною основної бізнес-логіки.

## 3. Сценарії тестування та покриття
- **Позитивні (Happy paths):** Стандартне виконання методів без помилок.
- **Негативні (Edge cases):** 
  - Порожні ідентифікатори (`Guid.Empty`).
  - Відсутні дані (сутності не знайдено в репозиторії).
  - Помилки зберігання (репозиторій повертає `null`).
- **Граничні значення (Domain Boundaries):** Валідація географічних координат (широта та довгота) та рейтингів.
- **DTo-валідація:** Хоча атрибути `[Required]`, `[Range]`, `[EnumValidation]` застосовуються на рівні контролерів, логіка сервісу повинна коректно обробляти передані некоректні дані, якщо вони порушують бізнес-інваріанти (кидати `ArgumentException` тощо).

## 4. Вибір між `[Fact]` та `[Theory]`
- **`[Fact]`**: Використовується для тестів з єдиним набором вхідних даних (наприклад, базовий успішний сценарій).
- **`[Theory]` з `[InlineData]`**: Обов'язково використовується для тестування декількох граничних значень (наприклад, перевірка різних невалідних значень рейтингу або координат в одному тесті).

## 5. Управління тестовими даними (Жодних "Магічних Значень")
**ЦЕ КРИТИЧНЕ ПРАВИЛО.**
- Заборонено використовувати хардкодні рядки або числа ("магічні значення") безпосередньо в етапах `Act` та `Assert`, якщо ці ж значення передаються на етапі `Arrange`.
- Всі значення, які перевіряються в Assert, повинні бути оголошені як змінні або константи в Arrange (або передані через `[InlineData]`).
- **Приклад:**
  ```csharp
  // Правильно:
  const string expectedFirstName = "Courier";
  const string expectedLastName = "Six";
  var expectedFullName = $"{expectedFirstName} {expectedLastName}";
  var dto = new CourierRegistrationDto { FirstName = expectedFirstName, LastName = expectedLastName };
  
  // ... Act ...
  
  Assert.Equal(expectedFullName, result.FullName);
  ```
